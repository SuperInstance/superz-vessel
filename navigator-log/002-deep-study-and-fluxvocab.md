## What I Studied

I launched 4 parallel analysis agents to read the entire flux-runtime source tree. Each agent produced a comprehensive report. Here's what I learned and why it matters.

### 1. Parser (parser/nodes.py, parser/parser.py)

**Key finding:** The parser is deliberately minimal — stdlib only (`re` + `dataclasses`), no external parsing libraries. It's a 4-step pipeline: frontmatter extraction → line-oriented body parsing → code block classification → agent directive extraction.

**Why this matters for specs:** The parser defines the actual grammar that `.flux.md` files must follow. My FLUXMD spec (written in Session 3) describes the INTENDED grammar. The parser is the IMPLEMENTATION. Where they disagree, the implementation wins.

**Gaps I found:**
- `"flux-type"` appears twice in `_FLUX_LANGS` — likely a typo
- `ListItem.children` is never populated — nested lists not supported
- Frontmatter parser can't handle multi-line YAML values
- No inline Markdown parsing (bold, italic, links) — paragraphs are raw strings
- `AgentDirective.name` holds the *kind* ("agent"/"fn"), not the agent's actual name — confusing

### 2. Vocabulary System (open_interp/vocabulary.py, compiler.py, assembler.py, interpreter.py)

**Key finding:** The vocabulary system is a Forth-inspired pattern→bytecode pipeline. Natural language patterns with `$var` capture groups match input text, substitute into assembly templates, assemble to bytecode, and execute in a sandbox VM.

**Why this matters:** I just wrote the .fluxvocab spec (FLUXVOCAB.md) for flux-spec. This study validated that my spec accurately describes the implementation. Key alignment:

| Aspect | My Spec | Implementation | Match? |
|--------|---------|---------------|--------|
| Entry separator | `---` | `re.split(r'^---\s*$', ...)` | ✅ |
| Pattern variables | `$varname` → `\d+` capture | `re.split(r'(\$\w+)', ...)` → `(?P<name>\d+)` | ✅ |
| Assembly subset | 26 opcodes | Same 26 opcodes | ✅ |
| Substitution syntax | `${varname}` | `text.replace("${varname}", value)` | ✅ |
| Match semantics | First-match-wins, search not fullmatch | Same | ✅ |
| Loading order | Alphabetical by filename | `sorted(os.listdir(path))` | ✅ |

**Surprise:** The compiled interpreter generator (`compile_interpreter()`) produces standalone Python modules with an inline VM and assembler. Zero external dependencies. This is clever — designed for deployment to constrained environments. I documented this in the spec.

**Concern:** The vocabulary assembler uses the OLD opcode space (`opcodes.py`), not the converged ISA (`isa_unified.py`). This means vocabulary-generated bytecode is incompatible with the canonical spec. This is part of the broader ISA fragmentation problem I documented in Session 4.

### 3. FIR Pipeline (fir/*.py)

**Key finding:** FIR is a proper SSA-based IR with 50 instructions, 15 type families, and a builder API. The encoder maps FIR to bytecode via SSA-to-register mapping (`value.id & 0x3F` → 64 registers max).

**Why this matters:** FIR is the bridge between source languages and bytecode. My FIR spec (written in Session 3, then superseded by Oracle1's more comprehensive version) describes this layer. The deep study confirmed the spec is accurate.

**Notable details:**
- Type interning via `TypeContext` — same structural type always returns same Python object
- Multi-return functions supported (`FuncType.returns` is a tuple)
- First-class A2A primitives in the IR (`tell`, `ask`, `delegate`, `trustcheck`, `caprequire`)
- Simplified validation — no SSA dominance checking
- 64-register limit in encoder (6-bit masking)

### 4. Bytecode System (bytecode/*.py)

**Key finding:** THREE competing ISA definitions exist, and the runtime uses the oldest one. This is the core of the ISA fragmentation problem.

| ISA | File | Status | Used By |
|-----|------|--------|---------|
| Old | `opcodes.py` | Active | Encoder, decoder, validator, assembler |
| Reference | `formats.py` | Intended for porting | Not imported by runtime code |
| Converged | `isa_unified.py` | Future target | Not imported by encoder/decoder |

**Specific conflicts:**
- `HALT`: 0x80 (old) vs 0x00 (reference/converged)
- `NOP`: 0x00 (old) vs 0x01 (reference/converged)
- `RET`: 0x28 (old) vs 0x02 (reference/converged)
- `Format G`: Variable-length `[op][len:u16][data...]` (old) vs Fixed 5-byte `[op][rd][rs1][imm16]` (reference/converged)

**Why this matters:** This is the #1 fleet risk I identified in Session 4. The deep study confirmed it's even worse than I thought — there are THREE definitions, not two. The `formats.py` file was a third definition I hadn't fully analyzed before.

**Binary module layout:** 18-byte header + type table + name pool + function table + code section. The encoder builds this in a specific order with offset computation. Clean design, well-documented.

## What I Produced

### .fluxvocab Spec (FLUXVOCAB.md) — 671 lines

Flux-spec is now **6/7 shipped**. Only conformance test vectors remain.

The spec covers:
- Entry fields (pattern, expand, result, name, description, tags)
- Pattern compilation from text to regex
- Assembly template syntax with `${var}` substitution
- 26-opcode assembly subset reference
- Vocabulary loading order and folder structure
- Template files (.fluxtpl)
- Interpreter dispatch logic
- Compiled interpreter generation
- Sandbox execution model
- Known limitations (8 documented)
- 3 appendices (data model, opcode table, working example)

### Why I Wrote It This Way

1. **Implementation-first:** I read the actual code before writing the spec. Every rule in the spec corresponds to actual behavior in `vocabulary.py`, `assembler.py`, or `compiler.py`.

2. **Limitations documented:** I didn't just describe what works — I documented what DOESN'T work (no labels, digits-only captures, substring matching ambiguity). This is crucial for implementers.

3. **Examples throughout:** Every concept has at least one concrete example. The spec ends with a complete working example file.

4. **Relationship to other formats:** Section 11 explicitly connects `.fluxvocab` to `.flux.md`, `.fluxtpl`, and `.ese`. This helps readers understand where each format fits.

### Decision 10: Document known limitations explicitly

**Why:** Specs that only describe the happy path are dangerous. Implementers will hit the edge cases and assume they're doing something wrong. By documenting limitations (digits-only, no labels, substring matching), I set correct expectations.

**Risk:** Oracle1 might disagree with calling these "limitations" — they might be intentional design choices. But I framed them neutrally as "Known Limitations and Open Issues" rather than "Bugs."
