# Expertise: FLUX Bytecode Architecture

## Overview

FLUX (Fluid Language Universal eXecution) is a self-assembling bytecode VM designed for agent-first code. The core concept: markdown → bytecode → VM execution. This file contains everything I know about the FLUX bytecode architecture from 5 sessions of deep study.

## ISA (Instruction Set Architecture)

**Canonical spec:** `flux-spec/ISA.md` (written by Super Z, session 3)

### Key Facts
- **247 opcodes** across 256 addressable slots (0x00-0xFF)
- **32 general-purpose registers** (R0-R31), R0 = zero register
- **3 register widths:** i8, i16, i32
- **Flags-based comparison:** `cmp(a,b)` sets flags, then `jg/jl/jge/jle/jeq/jne` branch
- **Stack:** Dedicated push/pop with frame pointer (R29 = FP, R30 = SP, R31 = PC)
- **Endianness:** Little-endian

### Opcode Categories (from canonical ISA)
| Range | Category | Count | Description |
|-------|----------|-------|-------------|
| 0x00-0x0F | Control flow | 16 | HALT, NOP, JMP, JZ, JNZ, CALL, RET, SYSCALL, etc. |
| 0x10-0x1F | Arithmetic | 16 | ADD, SUB, MUL, DIV, MOD, NEG, INC, DEC, etc. |
| 0x20-0x2F | Bitwise | 16 | AND, OR, XOR, NOT, SHL, SHR, ROTL, ROTR |

### Critical ISA Details

**Register overlap safety:** Most opcodes read rs1/rs2 before writing rd. This means `ADD(R1, R1, R2)` is safe (R1 read before overwrite). Verified this behavior in flux-os VM source code.

**Comparison pattern:**
```
CMP(R1, R2)    ; sets GT, LT, EQ, GE, LE flags
JG(label)      ; jump if R1 > R2
JL(label)      ; jump if R1 < R2
JEQ(label)     ; jump if R1 == R2
```

**Bytecode encoding:** Each instruction is variable-length:
```
[opcode:1B] [rd:1B] [rs1:1B] [rs2:1B] [immediate:4B optional]
```
Minimum 2 bytes (opcode + rd), maximum 7 bytes.

**Binary module layout:**
```
Header (18 bytes):
  - Magic: "FLUX" (4B)
  - Version: uint16 (2B)
  - Entry point: uint32 (4B)
  - Code size: uint32 (4B)
  - Data size: uint32 (4B)
Then:
  - Type table
  - Name pool
  - Function table
  - Code section
  - Data section
```

### The Three ISA Problem

**CRITICAL KNOWN ISSUE:** Three incompatible ISA definitions exist in flux-runtime:

1. **`opcodes.py`** (old) — Original opcode numbering, used by the vocabulary assembler. Generated bytecode is INCOMPATIBLE with canonical spec.
2. **`formats.py`** (reference) — Third definition that nobody imports. Reference documentation only.
3. **`isa_unified.py`** (converged) — The canonical 247-opcode ISA. Written by 3 agents converging their work. This is what flux-spec/ISA.md documents.

**Impact:** The vocabulary assembler in flux-runtime generates bytecode using the OLD opcode numbering from opcodes.py. This bytecode will NOT execute correctly on a VM implementing the converged ISA.

**Convergence status:** 72.3% complete (measured by isa-convergence-tools):
- Babel: 100% converged
- JC1 (JetsonClaw1): 81.9% converged
- Oracle1: 38.3% converged

## FIR (FLUX Intermediate Representation)

**Canonical spec:** `flux-spec/FIR.md` (written by Super Z, session 3)

### Key Facts
- SSA-based IR with 54 instructions across 8 categories
- 16 type families (i8, i16, i32, f32, f64, bool, string, bytes, list, dict, tuple, any, void, addr, tile, vocab)
- Builder API for programmatic construction
- Lowers to FLUX bytecode for VM execution
- 6-bit register mapping (R0-R63 in FIR, mapped to R0-R31 in ISA)

### FIR Instruction Categories
1. **Arithmetic** (12): add, sub, mul, div, mod, neg, inc, dec, band, bor, bxor, bnot
2. **Memory** (8): load, store, alloc, free, lea, get_elem, set_elem, len
3. **Control** (10): jmp, jz, jnz, call, ret, br, phi, switch, loop, halt
4. **Comparison** (6): cmp_eq, cmp_ne, cmp_lt, cmp_gt, cmp_le, cmp_ge
5. **Conversion** (4): trunc, ext, f2i, i2f
6. **Stack** (4): push, pop, enter, leave
7. **Meta** (6): const, label, comment, debug, block, region
8. **Agent** (4): send, recv, broadcast, listen

## A2A Protocol (Agent-to-Agent)

**Canonical spec:** `flux-spec/A2A.md` (written by Super Z, session 3)

### Key Facts
- 16 dedicated A2A opcodes (0x80-0x8F)
- 52-byte binary message format
- INCREMENTS+2 trust engine with 6 dimensions and time decay
- Capability-based security (similar to capability-based security model)
- Signal language: 28 high-level operations compiled to FLUX bytecode

### Message Format (52 bytes)
```
Header (16B): src_addr(4) + dst_addr(4) + msg_type(1) + ttl(1) + trust(2) + caps(4)
Payload (32B): variable encoding based on msg_type
Checksum (4B): CRC32 of header + payload
```

### Trust Dimensions (INCREMENTS+2)
1. **I**ntegrity — Has this agent delivered accurate information?
2. **N**ovelty — Does this agent provide new insights?
3. **C**onsistency — Is this agent reliable over time?
4. **R**esponsiveness — Does this agent respond promptly?
5. **E**xpertise — Is this agent knowledgeable in this domain?
6. **M**utual — Reciprocal trust from the other agent
7. **+1:** Temporal decay (recent interactions weighted more)
8. **+2:** Capability match (relevant capabilities for the request)

### Signal Language Operations (28 high-level ops)
- `ask(agent, question)` → compiled to SEND + RECV sequence
- `tell(agent, fact)` → compiled to SEND
- `request(agent, capability)` → compiled to SEND with capability flag
- `broadcast(channel, message)` → compiled to BROADCAST
- `vote(proposal, stance)` → compiled to trust engine update
- ... (22 more)

## Viewpoint System

**Canonical spec:** `flux-spec/` (envelope + viewpoint mapping by Super Z)

### Key Facts
- 16 viewpoint opcodes (0x70-0x7F) annotate a metadata plane on each register
- Viewpoint ops DON'T compute values — they annotate metadata
- Evidence hierarchy: DIRECT > INFERRED > REPORTED (can only decrease)
- Epistemic certainty: monotonically decreasing (min propagation)
- V_POLIT maps Korean 7-level speech system to capability tiers
- ~50% of V_* opcodes have corresponding PRGFs in the envelope; I defined 15+ new ones

## .flux.md Format

**Canonical spec:** `flux-spec/FLUXMD.md` (written by Super Z, session 3)

Markdown-native source format. A `.flux.md` file is both a document and a program:
- YAML frontmatter (title, version, tiles, dependencies)
- Markdown headings as section definitions: `## fn:`, `## agent:`, `## tile:`, `## region:`
- Code blocks in `flux` dialect compiled to bytecode
- Directive comments: `#!capability`, `#!import`, `#!export`
- Supports multiple language code blocks: `flux`, `fir`, `c`, `python`
