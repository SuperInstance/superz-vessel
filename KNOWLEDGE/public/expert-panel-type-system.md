## FLUX Type System — Current State

Before the debate begins, we establish the ground truth of what exists today.

### FIR (Flux Intermediate Representation)
- **54 SSA instructions** across 8 categories
- **16 type families** listed but not formally defined: `i8, i16, i32, f32, f64, bool, string, bytes, list, dict, tuple, any, void, addr, tile, vocab`
- **TypeField enum** references: `scalar_int, scalar_float, vector_int, vector_float`
- **3 data widths:** `i8, i16, i32`
- No type checking pass exists in any compiler pipeline

### FUTS (Flux Unified Type System)
- Mentioned in `flux-a2a-prototype` (~938 LOC in `types.py`)
- **12 `FluxBaseType` kinds:** scalar, vector, matrix, function, agent, region, bytecode, capability, quantum, stream, constraint, union
- **Constraints:** language-specific qualifiers (classifier, kasus, honorific, vibhakti)
- **Confidence score:** `0.0–1.0` attached to every typed value
- **Quantum type state:** superposition of possible types, collapsed on observation
- **Not integrated** with any FIR or ISA component

### Runtime Reality
- **Primary runtime** (`flux-py`): Pure Python, dynamically typed, no type checking in bytecode
- **Production runtime** (`flux-core`): Rust, no type information in bytecode format
- **7+ runtime implementations:** Rust, Zig, JavaScript, Python, Java, CUDA, WASM
- **Binary module layout** includes a "type table" section — currently empty/unused

### Confidence-Aware Operations
- **16 dedicated opcodes** at `0x60–0x6F` in the converged ISA
- Parallel confidence register file: every register `R_i` has `C_i`
- `CONF_LD` / `CONF_ST`: Load/store confidence registers
- Confidence propagation semantics (from design docs):
  - `CONF_ADD`: `rd = rs1 + rs2, crd = min(crs1, crs2)`
  - `CONF_MUL`: `rd = rs1 * rs2, crd = crs1 * crs2 * (1 - ε)`
  - `CONF_MIN`: `crd = min(crs1, crs2)`
  - `STRIPCF`: Strip confidence from next N operations

## Debate 1: Should FLUX Bytecode Be Typed At All?

### Dr. Hindley — Theorist

Let me begin with the strongest possible claim: **untyped bytecode is a liability, not a feature.** The FLUX VM executes agent-generated code — code that may have been compiled by an AI, assembled from fragments across the fleet, or stitched together from vocabulary plugins written in different languages. Without a type system, the only defense against a category error is a runtime crash, and runtime crashes in agent systems are catastrophic because they don't just terminate a process — they terminate a reasoning chain.

Consider the formal argument. In the simply-typed lambda calculus, we have the typing rule for application:

```
Γ ⊢ e₁ : τ → σ    Γ ⊢ e₂ : τ
────────────────────────────────  (App)
      Γ ⊢ e₁ e₂ : σ
```

This rule guarantees that every function receives exactly the kind of argument it expects. Without it, `ADD(R1, R2)` where `R1` holds a string and `R2` holds a memory address produces undefined behavior. The FLUX ISA currently has no way to prevent this at compile time. The `0x20` (ADD) opcode operates on raw bit patterns. That was acceptable when bytecode was hand-written by a single author. It is not acceptable when bytecode is the product of multi-agent assembly.

The argument that "dynamic typing is sufficient" fails under composition. Type A and Type B may each be safe when used alone. But when Agent X produces a tile of type `vector_int` and Agent Y consumes it expecting `vector_float`, the composition silently corrupts data. This is the well-known "safe at rest, unsafe in composition" problem, and the only solution is a shared type discipline.

FLUX needs a type system because FLUX is a composition platform. The `.flux.md` format explicitly supports multi-language code blocks — `flux`, `fir`, `c`, `python` — and vocabulary plugins that may be written in any of them. A type system is the contract that makes this composition safe.

### Prof. Gradual — Pragmatist

I agree with Dr. Hindley's conclusion — FLUX bytecode should be typed — but I fundamentally disagree with the methodology. The question isn't "should it be typed?" but "can we afford to type it all at once?" and the answer is clearly no.

The current FLUX ecosystem has seven runtime implementations, 247 opcodes (many still speculative), and zero deployed bytecode with type annotations. The flux-a2a-prototype has a FUTS implementation, but it's a 938-line Python module that hasn't been integrated with anything. If we mandate Hindley-Milner type inference across all FIR programs tomorrow, we don't get safety — we get a migration crisis that makes the ISA convergence look trivial.

Gradual typing, as formalized by Siek and Taha (2006), provides the answer. The key insight is the **consistency relation** `τ₁ ~ τ₂` rather than type equality `τ₁ = τ₂`:

```
dyn ~ τ     for all τ         (G-Static)
τ ~ dyn     for all τ         (G-Dynamic)
int ~ int                      (G-Refl)
```

Under gradual typing, FLUX bytecode can be **optionally typed**. A vocabulary plugin written in Rust can declare full type signatures. A dynamically loaded Signal program compiled by an AI can carry `dyn` for every register. The VM enforces type consistency at boundaries between typed and untyped code, inserting casts where needed. TypeScript proved this works at scale — millions of `.js` files coexist with millions of `.ts` files, and the type system catches errors at the seams without forcing adoption.

For FLUX specifically, I propose: the type table in the binary module layout becomes optional. If present, the VM performs type checking on entry to each function. If absent, the VM operates in fully dynamic mode — exactly as it does today. This is a **non-breaking change** that enables type adoption at the pace of each vocabulary author.

### Eng. Linear — Resource Theorist

Both of you are arguing about value types — "is this an int or a float?" — and missing the more important dimension: **resource types**. FLUX agents operate in resource-constrained environments: limited memory regions, bounded energy budgets, network channels with capacity limits. A type system that tracks only values but not resources is half a type system.

Consider FLUX's tile system. Tiles are computational units — a `tile_new` creates a bounded workspace, `tile_get` and `tile_put` move data in and out, and eventually the tile should be freed. Currently there's nothing preventing:
1. Double-free of a tile (use after free in agent context)
2. Memory leak (tile allocated but never freed)
3. Cross-tile aliasing (two agents pointing at the same tile, one frees it)

Linear types, as formalized by Girard (1987) and practicalized by Rust's ownership system, solve all three:

```
Γ ⊢ e : A    A ⊸ B    Γ ⊢ e' : B → C
────────────────────────────────────────  (Linear-App)
         Γ ⊢ e; e' : C

Γ, x : A ⊢ e : B
────────────────────  (Linear-Abs)     (x used exactly once in e)
     Γ ⊢ λx. e : A ⊸ B
```

The linear arrow `⊸` (lollipop) guarantees that each resource is used **exactly once**. For FLUX, this means every tile allocation has exactly one corresponding free. Every channel send has exactly one receive. Every capability is exercised exactly once.

FLUX bytecode should be typed because agents need **resource safety** more than they need value safety. A category error corrupts one computation; a resource leak corrupts the entire agent ecosystem.

### Dr. Confidence — Uncertainty Specialist

My three colleagues are all operating under the classical assumption that types describe **sets of values**. A type `int32` denotes the set `{ -2³¹, ..., 2³¹-1 }`. But in FLUX's domain — AI agents reasoning about uncertain data — this assumption is fundamentally wrong.

When a sensor reports temperature as `23°C`, that's not `int32`. It's a **probabilistic value**: a distribution centered at 23 with some variance, produced by a measurement process with known error characteristics. When two agents share a value over the A2A protocol, the confidence degrades — `CONF_MUL` models this with `crd = crs1 * crs2 * (1-ε)`, where ε captures transmission uncertainty.

I propose that in FLUX, a type is not a set but a **probability distribution over a set**:

```
τ ::= { x ∈ S | P(x) }       -- probabilistic type: distribution P over set S
```

So `conf_int32(μ=23, σ=2, c=0.85)` means: the value is an int32 drawn from a distribution with mean 23, standard deviation 2, and the agent's confidence in this model is 0.85. This isn't refinement typing — refinement types add predicates `{ x : int | x > 0 }`. This is **distributional typing**, which is strictly more expressive because it captures not just whether a value satisfies a constraint, but how *confident* we are that it does.

The confidence register file already exists in the converged ISA (opcodes 0x60–0x6F). This IS a type system — it's just one that nobody has recognized as such. Every `CONF_ADD`, `CONF_MUL`, `CONF_MIN` is a type-level operation that propagates uncertainty through the computation. The tragedy is that it's implemented as a side channel rather than a first-class type qualifier.

**Yes, FLUX bytecode should be typed — and the type system should carry confidence bounds as a first-class component of every type.**

## Debate 2: What Type System Foundation?

### Dr. Hindley — Theorist

The foundation should be **System Fω** (higher-order polymorphic lambda calculus) extended with **row types** for records and **kind polymorphism** for vocabulary extensibility. Let me sketch the formal system.

**Kinds:**

```
κ ::= *           -- base kind (types)
    | κ → κ       -- kind of type constructors
    | Row         -- kind of row labels
```

**Types:**

```
τ ::= α                      -- type variable
    | τ → τ                  -- function type
    | ∀α:κ. τ                -- universal type (parametric polymorphism)
    | Λα:κ. τ                -- type abstraction
    | τ [κ]                  -- type application
    | { row(ρ) }             -- record type with row ρ
    | τ | τ                  -- sum type (algebraic variant)
    | τ × τ                  -- product type
    | [τ]                    -- vector type (fixed or dynamic length)
    | Tile(τ)                -- tile containing values of type τ
    | Vocab(Σ)               -- vocabulary with signature Σ
```

**Row Types** (for extensible records):

```
ρ ::= ε                      -- empty row
    | l : τ | ρ              -- label l maps to type τ, rest is ρ
    | ρ ⊕ ρ'                 -- row concatenation
```

This gives us parametric polymorphism (a single `MAP` tile works on `[int]`, `[float]`, `[addr]`), extensible records (vocabulary plugins add new fields without breaking existing code), and algebraic data types (sum types for FIR's 16 type families). System Fω is decidable, well-understood, and has efficient type inference via Algorithm W + generalization.

The key requirement for FLUX is that this type system is **expressive enough to type the existing ISA without modification**. Every opcode in the converged spec should have a type signature in this system. For example:

```
ADD : ∀w. i(w) → i(w) → i(w)              -- w ∈ {8, 16, 32}
LOAD : ∀w τ. addr → mem(w, τ) → τ          -- load from typed memory
TILE_NEW : ∀τ. nat → Tile(τ)               -- allocate tile of n elements
SEND : ∀τ. agent → τ → io(())              -- send typed value to agent
CONF_ADD : ∀w. i(w) × c → i(w) × c → i(w) × c  -- propagate confidence
```

### Prof. Gradual — Pragmatist

Dr. Hindley's proposal is beautiful — in the same way that a formal verification proof of a "Hello World" program is beautiful. It's over-engineered by a factor of ten for FLUX's current needs.

Here's what I'd actually build:

**Step 1: The Dynamic Default.** Every register in FLUX bytecode has type `dyn` by default. No type table in the binary module = all `dyn`. Zero overhead, zero migration cost.

**Step 2: Gradual Annotations.** Add an optional type annotation to register declarations:

```
;; Untyped (current behavior)
MOVI R1, 42
ADD  R2, R1, R3        ; no checking, runtime error on type mismatch

;; Typed (new behavior, opt-in)
;; @type R1: i32, R2: i32, R3: i32
MOVI R1, 42
ADD  R2, R1, R3        ; checked: all i32 ✓
```

**Step 3: Sealing at Boundaries.** When a typed tile calls an untyped vocabulary, the output is automatically `dyn`. When an untyped tile calls a typed vocabulary, the input is checked at runtime:

```
;; tile_A.flux (typed)
;; @sig compute(x: i32, y: f32) -> f32

;; tile_B.flux (untyped)
;; No annotations — works fine

;; Composition (boundary checking)
tile_A.compute(tile_B.result)  ; tile_B.result is dyn → runtime cast check
```

**Step 4: Progressive Inference.** As more tiles adopt types, the inference engine fills in gaps. But inference is **never mandatory**. A tile author can always write `dyn` and move on.

This is the TypeScript model: `any` everywhere by default, explicit annotations where they help, inference where it's unambiguous. The formal foundation is **the gradual type system of Siek & Taha**, which has a well-defined cast calculus:

```
⟨e, dyn⟩ → ⟨e', τ⟩   :   insert cast (dyn → τ)
⟨e, τ⟩ → ⟨e', dyn⟩   :   insert cast (τ → dyn)
⟨e, τ₁⟩ → ⟨e', τ₂⟩   :   insert cast (τ₁ → τ₂) if τ₁ ~ τ₂, error otherwise
```

The type system grows with the ecosystem. Today it's `dyn` everywhere. Next year it's `i32` and `f32` on the hot path. In five years, maybe Dr. Hindley's System Fω — but only if the ecosystem demands it.

### Eng. Linear — Resource Theorist

Both proposals miss the resource dimension. Let me be concrete about what FLUX needs.

**Affine Type System** (resources used at most once, not exactly once):

```
Γ ⊢ alloc(n) : Region(n)           -- allocate n bytes
Γ, r : Region(n) ⊢ use(r) : τ      -- use region (consumes it)
Γ, r : Region(n) ⊢ free(r) : ()    -- free region (consumes it)
```

I say *affine* rather than *linear* because strict linearity (exactly-once use) is too rigid for FLUX. Some resources should be droppable — if a tile allocates a region and then the tile errors out, the region should be garbage collected, not leaked AND not requiring explicit free. Rust solved this with Drop traits. FLUX can solve it with a modified affine system:

```
A ⊳ B     -- affine arrow: A is used AT MOST once in B
A ⊸ B     -- linear arrow: A is used EXACTLY once in B
A → B     -- unrestricted arrow: A can be used any number of times (normal)
```

**Practical mapping to FLUX:**

| FLUX Concept | Linear/Affine Type | Enforcement |
|---|---|---|
| Memory region (`ALLOC`/`FREE`) | `Region(n) ⊳ τ` | Compiler inserts `free` on all paths |
| Tile (`TILE_NEW`/`TILE_PUT`/`TILE_GET`) | `Tile(τ) ⊳ ()` | Tile has exactly one owner |
| A2A channel (`CHANNEL`/`SEND`/`RECV`) | `Chan(τ) ⊳ τ` | Send consumes the channel handle |
| Capability (`#!capability`) | `Cap(p) ⊸ τ` | Capability exercised exactly once or revoked |
| Confidence register | `c : [0,1]` (unrestricted) | Confidence is copyable, not a resource |

The beauty of affine types for FLUX: **the existing ISA doesn't need to change**. The type checker runs on the FIR layer, inserting `FREE` instructions where affine analysis proves they're needed. The bytecode is the same — it's just provably resource-safe.

### Dr. Confidence — Uncertainty Specialist

I propose a **probabilistic type system** where types are distributions, not sets. This is the natural foundation for FLUX because confidence is already first-class in the ISA.

**Core idea:** A type `𝒯` is a pair `(S, P)` where `S` is a set of values and `P` is a probability measure over `S`:

```
𝒯 ::= (S, P(x))                   -- probabilistic type

-- Shorthands:
i32_c(μ, σ, c) = ({ -2³¹...2³¹-1 }, N(μ, σ²) · I(P(x) ≥ c))
                  -- int32 with Gaussian belief, confidence threshold c

f32_c(μ, σ, c) = ({ IEEE-754 floats }, N(μ, σ²) · I(P(x) ≥ c))

dyn_c(c)       = ({ all values }, Uniform · I(c))
                  -- complete uncertainty, just a confidence level
```

**Type operations** correspond to confidence register operations:

```
-- CONF_ADD semantics:
i32_c(μ₁, σ₁, c₁) +ᶜ i32_c(μ₂, σ₂, c₂) = i32_c(μ₁+μ₂, √(σ₁²+σ₂²), min(c₁, c₂))

-- CONF_MUL semantics:
i32_c(μ₁, σ₁, c₁) *ᶜ i32_c(μ₂, σ₂, c₂) = i32_c(μ₁·μ₂, f(σ₁,σ₂,μ₁,μ₂), c₁·c₂·(1-ε))

-- CONF_MIN semantics (for branching):
min_conf(c₁, c₂) = min(c₁, c₂)        -- conservative: belief is min of constituents

-- CONF_SET semantics (for sensor input):
set_conf(v, c) = i32_c(v, 0, c)       -- known value with stated confidence
```

**Subtyping** is replaced by a **confidence-ordered relation:**

```
𝒯₁ ≤_c 𝒯₂    iff    S₁ ⊆ S₂  ∧  c₁ ≥ c₂
```

An `i32_c(μ, σ, 0.95)` is a subtype of `i32_c(μ, σ, 0.80)` — higher confidence is more specific. This gives us a natural subsumption rule:

```
Γ ⊢ e : τ_c(0.95)
──────────────────────  (Confidence-Sub)
Γ ⊢ e : τ_c(0.80)
```

This is the type system FLUX was designed for. The confidence register file is the hardware manifestation of the type qualifier. The `CONF_*` opcodes are type operations. FUTS just needs to formalize what's already implicit in the ISA.

## Debate 3: The Confidence Register File as Type System

### Dr. Confidence — Uncertainty Specialist

Let me make the argument explicit: **FLUX already has a type system, and it lives in the confidence register file.**

The current design pairs every data register `R_i` with a confidence register `C_i`. When you execute `ADD R1, R2, R3`, the hardware (or VM) simultaneously computes `C1 = min(C2, C3)`. This is not a side effect — it's a type propagation rule.

In traditional type theory, the typing judgment is `Γ ⊢ e : τ` — under context Γ, expression e has type τ. In FLUX, the typing judgment is:

```
Γ ⊢ e : (v, c)
```

where `v` is the value and `c ∈ [0, 1]` is the confidence. Every SSA value in FIR carries both components. The confidence register file is the **runtime representation of the type qualifier**.

Consider the formal correspondence:

| Traditional Type Theory | FLUX Confidence System |
|---|---|
| `Γ ⊢ e : int` | `Γ ⊢ e : (v, 1.0)` where v ∈ {-2³¹...2³¹-1} |
| `Γ ⊢ e : int` (from subtyping) | `Γ ⊢ e : (v, 0.95)` (high confidence int) |
| `Γ ⊢ e : dyn` | `Γ ⊢ e : (v, 0.0)` (no confidence = fully dynamic) |
| `cast(e, τ₁ → τ₂)` | `CONF_ADJUST(e, c_new)` |
| Type variable α | `α_c` where c is the confidence bound |
| Intersection type `τ₁ ∧ τ₂` | `merge(c₁, c₂)` — union of beliefs |

The `CONF_MUL` rule — `crd = crs1 * crs2 * (1-ε)` — is the type-level rule for **composition of uncertain values**. When you multiply two measurements, the uncertainty in each compounds, plus there's a small transmission loss ε. This is the probabilistic analog of the function composition rule:

```
Γ ⊢ f : τ_c₁ → σ_c₂     Γ ⊢ x : τ_c₁
───────────────────────────────────────────
         Γ ⊢ f(x) : σ_{c₂ · (1-ε)}
```

The `STRIPCF` opcode (strip confidence from next N ops) is the **dynamic-to-static cast**. It says "I know what I'm doing — treat these values as fully confident." This is the analog of TypeScript's `as` keyword or Haskell's `unsafeCoerce`.

The confidence register file is not a debugging aid or a research curiosity. It IS the type system. The question is whether we formalize it or let it remain implicit.

### Dr. Hindley — Theorist

Dr. Confidence's insight is genuinely original, and I want to engage with it seriously. The mapping from confidence to refinement types is tight:

A refinement type `{ x : int | P(x) }` attaches a logical predicate P to a base type. Dr. Confidence is proposing `{ x : int | conf(x) ≥ c }` where `conf(x)` is the confidence that x is correct. This is a specific instance of refinement typing where the refinement predicate measures uncertainty.

However, there's a crucial distinction. In classical refinement types, the refinement is **boolean**: the value either satisfies the predicate or it doesn't. In Dr. Confidence's system, the confidence is **continuous**: it's not "does this value satisfy P?" but "how sure are we that this value satisfies P?" This pushes us from classical logic into **probabilistic logic**, which is a very different formal foundation.

I'd formalize the connection as follows. Let `𝒞(τ)` denote "type τ with confidence annotation":

```
𝒞(τ) ::= τ{c}              where c ∈ [0, 1]

Subtyping: τ{c₁} <: τ{c₂}  iff  c₁ ≥ c₂

Product:   τ₁{c₁} × τ₂{c₂} = (τ₁ × τ₂){min(c₁, c₂)}

Function:  (τ₁{c₁} → τ₂{c₂})(c_op) = τ₁{c₁} → τ₂{c₂ · c_op}
```

The `c_op` parameter captures the confidence cost of the operation — this is exactly the `(1-ε)` in `CONF_MUL`.

I'd adopt this as a **type qualifier** in my System Fω proposal, making confidence a second dimension orthogonal to the value type:

```
Qualified Type:  τ @ c           -- value type τ, confidence c

Full FLUX type:  (BaseType × Width × Confidence)
                 = (int × i32 × 0.85)  =  int32 with 85% confidence
```

This is elegant and composable. But I'd insist on one caveat: confidence alone is not sufficient. You still need the structural type system (functions, products, sums, vectors) to express the shape of data. Confidence is the qualifier; you need the base types to qualify.

### Prof. Gradual — Pragmatist

This is where things get interesting. I've been advocating for gradual typing — dynamic by default, static when you opt in. Dr. Confidence's confidence system gives me the **bridge** I need.

Consider the correspondence:

| Gradual Typing Concept | Confidence System Analog |
|---|---|
| `dyn` (fully dynamic) | `c = 0.0` (zero confidence) |
| `int` (fully static) | `c = 1.0` (full confidence) |
| Type cast (boundary check) | `CONF_ADJUST` (adjust confidence at boundary) |
| Consistency relation `τ₁ ~ τ₂` | Confidence compatibility: `|c₁ - c₂| < threshold` |
| Type error (incompatible types) | Confidence breach: `c_result < c_threshold` |

Under this mapping, **gradual typing IS confidence typing**. The `dyn` type is just "zero confidence." A type cast is just a confidence adjustment. The entire gradual typing theory — the blame calculus of Wadler and Findler — maps onto confidence propagation.

This means we don't need to choose between gradual typing and confidence typing. They're the same thing, viewed from different angles. The implementation strategy is:

1. **Every register has type `(τ, c)`** where `τ` is the structural type and `c ∈ [0, 1]` is the confidence
2. **Default:** `τ = dyn, c = 0.0` — fully untyped, zero confidence (current behavior)
3. **Opt-in:** Vocabulary authors specify `τ` and/or `c` as they see fit
4. **Boundary:** When a `(τ₁, c₁)` value flows to a `(τ₂, c₂)` context:
   - If `τ₁ ~ τ₂` and `c₁ ≥ c₂`: no cast needed, value flows freely
   - If `τ₁ ~ τ₂` and `c₁ < c₂`: insert confidence adjustment (CONF_ADJUST)
   - If `τ₁ !~ τ₂`: insert dynamic type check (cast) + confidence penalty

The beauty: **confidence propagation gives gradual typing its runtime semantics**. We don't need a separate blame tracking system — confidence IS the blame tracker. Low confidence means "we don't trust this value." The threshold where an agent rejects a value (`c < c_min`) IS the type error.

### Eng. Linear — Resource Theorist

I'll keep my response brief because the other three have covered the value-type dimension thoroughly. My point: **confidence and linear types are orthogonal dimensions, and both are needed.**

Linear types track **resource ownership** — "who owns this memory, and when is it freed?" Confidence types track **belief state** — "how much do we trust this value?" These are independent concerns:

```
Full FLUX Type = BaseType × Width × Confidence × Linearity

Examples:
- int32 @ 0.95 ⊳ owned     -- high-confidence int, I own the memory, must free it
- f32   @ 0.30 → shared    -- low-confidence float, shared reference, no ownership obligation
- addr  @ 0.80 ⊳ channel   -- confident address, linear channel handle, must send exactly once
- dyn   @ 0.00 → leaked    -- zero-confidence dynamic value, resource leak (type error!)
```

The last example is the killer insight: **resource leaks have a confidence dimension**. If a value has zero confidence, should the agent even bother tracking its resource ownership? Probably not — it's noise. But if a value has high confidence and is linear, a leak is a serious error.

I propose the joint qualifier system:

```
Q ::= c                 -- confidence only (default for values)
    | ⊳                 -- linear (owned, must consume exactly once)
    | →                 -- shared (unrestricted, can copy freely)
    | ⊳_c               -- linear with confidence c
    | →_c               -- shared with confidence c
```

The type checker enforces both dimensions simultaneously. A value that's linear AND low-confidence gets a warning: "You own this resource, but you're not confident in its contents. Are you sure?"

## Debate 4: FIR Type Families — Too Many? Too Few?

### Context

The current FIR specification lists 16 type families:

```
i8, i16, i32, f32, f64, bool, string, bytes, list, dict, tuple, any, void, addr, tile, vocab
```

The `TypeField` enum references four SIMD-related categories:

```
scalar_int, scalar_float, vector_int, vector_float
```

Neither list is formally defined. There's no type algebra, no subtyping rules, no size information beyond the integer widths.

### Dr. Hindley — Theorist

Sixteen type families is neither too many nor too few — it's **incoherent**. The list mixes levels of abstraction that shouldn't coexist in a flat enumeration:

1. **Width-qualified primitives:** `i8, i16, i32` (but not `i64`; `f32, f64` but not `f16`)
2. **Base types:** `bool, string, bytes`
3. **Container types:** `list, dict, tuple` (but no `set`, `queue`, `stack`)
4. **Meta types:** `any, void`
5. **FLUX-specific types:** `addr, tile, vocab`
6. **Width-qualified SIMD:** `scalar_int, scalar_float, vector_int, vector_float` (from TypeField, not in the 16 families)

This needs algebraic reconstruction. My proposal:

```
Base  ::= Int(w) | Float(w) | Bool | String | Bytes | Addr
       | Tile(τ) | Vocab(Σ) | Cap(p) | Region(n)

w     ::= 8 | 16 | 32 | 64

Container ::= List(τ) | Dict(τ_key, τ_val) | Tuple(τ₁, ..., τₙ)
            | Vec(w, τ, n) | Mat(w, τ, m, n)    -- SIMD vectors/matrices

Compound ::= Base | Container | Sum(variants) | Product(fields)

Meta  ::= Any | Void | Dyn
```

This gives us:
- **Parametric width:** `Int(w)` covers `i8, i16, i32, i64, i128` — no need to enumerate
- **Parametric containers:** `List(Int(32))` replaces a flat `list` with typed lists
- **SIMD as typed vectors:** `Vec(32, Float(32), 4)` is a 4-wide float32 vector — covers the TypeField categories
- **Algebraic data types:** `Sum` and `Product` replace ad-hoc variant handling
- **Agent-specific types:** `Tile`, `Vocab`, `Cap`, `Region` are first-class

The 16 families become **4 base categories × parametric instantiation** = infinite types from finite rules.

### Prof. Gradual — Pragmatist

Dr. Hindley's algebraic reconstruction is correct in principle but suicidal in practice. If we replace `i32` with `Int(32)`, every FIR program, every vocabulary, every test case needs rewriting. And we gain what? The ability to express `Int(128)` which no FLUX hardware supports?

My counter-proposal: **start with 4 base types and grow from there.**

```
Phase 1 (FUTS v0.1):
  - i32        (covers most arithmetic)
  - f32        (covers floating point)
  - dyn        (covers everything else)
  - void       (covers null/empty)

Phase 2 (FUTS v0.5):
  - i8, i16, i64, f64  (extended widths)
  - bool, bytes, string  (common data types)
  - addr       (pointers into memory)

Phase 3 (FUTS v1.0):
  - list(τ), dict(τ, τ), tuple(τ...)  (parametric containers)
  - tile(τ), vocab(Σ)  (FLUX-specific)

Phase 4 (FUTS v2.0):
  - vec(w, τ, n), mat(w, τ, m, n)  (SIMD)
  - region(n), cap(p)  (resource types)
  - Any → deprecated (use dyn instead)
```

The key insight: **the type system should be deployable incrementally**. FUTS v0.1 with 4 types is already better than the current zero types. Each phase adds types only when there's a concrete need — a vocabulary that requires parametric containers, a hardware target that needs SIMD vectors.

The 16 families should be **deprecated in favor of this progressive model**. Not because 16 is wrong, but because a flat enumeration doesn't compose. `list` should be `list(τ)`, not a standalone type.

### Eng. Linear — Resource Theorist

I'll focus on what's missing from the 16 families rather than what's there. The current list has zero resource types. For a system that manages memory regions, agent capabilities, A2A channels, and tile ownership, this is a critical gap.

Add to the 16 families:

```
Resource Types (add to FIR):
  - region(n)     -- memory region of n bytes (affine)
  - channel(τ)    -- A2A channel carrying values of type τ (linear)
  - cap(p)        -- capability with permission set p (linear)
  - handle(τ)     -- opaque handle to external resource (affine)
  - token(k)      -- authentication/bearer token (linear, uncloneable)
  - budget(e)     -- energy/time budget of e units (affine, consumed by ops)
```

The 16 + 6 resource types = 22 families. Not too many — each serves a distinct purpose. And unlike the current 16, the resource types have formal semantics (use-at-most-once, use-exactly-once) that the type checker can enforce.

I'd also argue that `tile` and `vocab` should be reclassified as resource types rather than value types. A tile is a bounded workspace — it has a size, an owner, and a lifetime. A vocabulary is a loaded code module — it has initialization costs, memory usage, and dependency relationships. Both should be affine at minimum.

### Dr. Confidence — Uncertainty Specialist

The 16 families are entirely about **what** a value is, not **how certain** we are about it. I'd add:

```
Uncertainty Types (qualify every existing type):
  - Certain(τ)     -- τ with confidence ≥ 0.95 (treated as fully typed)
  - Probable(τ)    -- τ with confidence ≥ 0.70 (warnings on use in critical paths)
  - Uncertain(τ)   -- τ with confidence ≥ 0.40 (errors in safety-critical contexts)
  - Unknown(τ)     -- τ with confidence < 0.40 (equivalent to dyn)
```

These aren't separate type families — they're **type constructors** that qualify any base type:

```
Certain(i32)   = i32 @ 0.95     -- "I'm very sure this is an i32"
Probable(f32)  = f32 @ 0.80     -- "Probably a float, might have noise"
Unknown(dyn)   = dyn @ 0.20     -- "Could be anything, low trust"
```

The 16 families stay as-is. The uncertainty constructors layer on top. This is the cheapest possible change — the type checker treats `Certain(τ)` as `τ` for all structural purposes, but adds confidence propagation. `Unknown(dyn)` is identical to the current `any` type. Zero breaking changes, maximum information gain.

## Debate 5: Multi-Language Type Interoperability

### Context

FLUX has 7+ runtime implementations: Rust, Zig, JavaScript, Python, Java, CUDA, WASM. A `.flux.md` file can contain code blocks in `flux`, `fir`, `c`, and `python`. Vocabularies are loaded at runtime from arbitrary sources. How do types map across this heterogeneity?

### Dr. Hindley — Theorist

The solution is a **canonical type representation** — a single, well-defined type algebra that serves as the interchange format between all languages. This is exactly what WebAssembly does with its `valtype`:

```
WASM valtype ::= i32 | i64 | f32 | f64 | v128 | funcref | externref
```

Every language maps its native types to WASM valtypes at the boundary. Inside WASM, everything is valtypes. Outside, each language reconstructs its native representation.

For FLUX, I propose the **FUTS Canonical Type** (FCT) as the interchange format:

```
FCT ::= Int(w)           -- w ∈ {8, 16, 32, 64}
     | Float(w)          -- w ∈ {16, 32, 64}
     | Bool
     | Bytes(n)          -- fixed-length byte array
     | String            -- UTF-8 encoded
     | Addr              -- opaque pointer (32-bit)
     | List(FCT)         -- homogeneous list
     | Tuple(FCT, ...)   -- heterogeneous tuple
     | Record(row(FCT))  -- labeled record
     | Sum(variant)      -- tagged union
     | Tile(FCT, n)      -- n-element tile of FCT values
     | Func(FCT..., FCT) -- function signature
     | Dyn               -- dynamically typed

Width encoding:
  w = 0 → 8-bit
  w = 1 → 16-bit
  w = 2 → 32-bit
  w = 3 → 64-bit
```

Binary encoding (for type tables and A2A messages):

```
[FCT tag: 1 byte] [width/payload: variable]

Tag 0x00 = Int(w)      payload: 1 byte width
Tag 0x01 = Float(w)    payload: 1 byte width
Tag 0x02 = Bool        payload: none
Tag 0x03 = Bytes(n)    payload: 2 bytes length
Tag 0x04 = String      payload: none
Tag 0x05 = Addr        payload: none
Tag 0x06 = List(FCT)   payload: 1 FCT (element type)
Tag 0x07 = Tuple       payload: 1 byte count + N FCTs
Tag 0x08 = Record      payload: row encoding
Tag 0x09 = Sum         payload: variant encoding
Tag 0x0A = Tile        payload: 1 FCT + 2 bytes count
Tag 0x0B = Func        payload: N FCTs + 1 FCT (return)
Tag 0x0C = Dyn         payload: none
```

Every runtime maps FCT ↔ native types. The type table in the binary module layout stores FCT-encoded types. A2A messages include FCT type annotations for their payloads. This is the WASM approach, and it works.

### Prof. Gradual — Pragmatist

A canonical type representation is necessary but not sufficient. The real problem is **dynamic interoperability** — what happens when two agents using different runtimes exchange values that neither has a static type for?

The answer is **JSON-like dynamic types** as the true interchange format:

```python
# Agent A (Rust runtime) sends a typed value:
#   struct Point { x: i32, y: i32 }
#   Encoded as: {"_futs_type": "record", "x": 42, "y": 17, "_conf": 0.95}

# Agent B (Python runtime) receives it:
#   No static type information → treated as dict with _futs_type metadata
#   Can use it as: point["x"], point["y"]
#   Confidence: point["_conf"] = 0.95
```

The `_futs_type` and `_conf` fields are optional metadata. If present, a typed runtime can reconstruct the full type. If absent, the value is `dyn`. This is exactly how TypeScript handles JSON from untyped APIs — you annotate the response type, but the wire format is plain JSON.

Mapping table for cross-language interop:

| FUTS Type | Python | Rust | JavaScript | CUDA | WASM |
|---|---|---|---|---|---|
| `Int(32)` | `int` | `i32` | `number` | `int` | `i32` |
| `Float(32)` | `float` | `f32` | `number` | `float` | `f32` |
| `Bool` | `bool` | `bool` | `boolean` | `N/A` | `i32` (0/1) |
| `String` | `str` | `String` | `string` | `N/A` | `(i32, i32)` ptr+len |
| `List(τ)` | `list` | `Vec<T>` | `Array` | `τ*` | `(i32, i32)` ptr+len |
| `Tuple(τ...)` | `tuple` | `(T,...)` | `Array` | `struct` | flat in memory |
| `Addr` | `int` | `u32` | `number` | `void*` | `i32` |
| `Dyn` | `Any` | `enum Value` | `any` | `void*` | `(i32, i32)` tag+ptr |
| `Tile(τ, n)` | `bytearray` | `[T; N]` | `ArrayBuffer` | `τ[N]` | `(i32, i32, i32)` |
| `Bytes(n)` | `bytes` | `[u8; N]` | `Uint8Array` | `unsigned char*` | `(i32, i32)` |

The key insight: **JSON with optional type metadata** is the universal interchange format. FCT (the canonical representation) is used in bytecode type tables and compiled interfaces. JSON is used at runtime for A2A messages and dynamic vocabulary loading. Both coexist.

### Eng. Linear — Resource Theorist

The cross-language mapping problem has a resource dimension that nobody has addressed. Consider:

1. **Rust runtime:** Has ownership and borrowing. A `Tile<i32>` in Rust is a `Vec<i32>` with a single owner.
2. **Python runtime:** Has garbage collection. A tile is a `bytearray` with shared references everywhere.
3. **CUDA runtime:** Has explicit memory management. A tile is a `device_ptr` that must be explicitly `cudaFree`'d.
4. **WASM runtime:** Has linear memory. A tile is an offset + length in linear memory.

When a tile moves from Rust to Python, who owns it? When CUDA sends a tile to WASM, who frees the GPU memory? The current system has **no answer** to these questions.

Linear types solve this. At the boundary:

```
Rust → Python:
  Tile(τ, n) ⊳ τ    (linear, owned in Rust)
  → transfer ownership to Python's GC
  → Rust's type system knows the tile is gone (linear consumed)
  → Python's GC tracks it from here

CUDA → WASM:
  Tile(τ, n) ⊳ τ    (linear, owned on GPU)
  → cudaMemcpy to host
  → cudaFree the GPU copy
  → transfer ownership to WASM's linear memory
  → WASM allocates in linear memory, tracks via linear type
```

The linear type annotation travels with the value across the A2A message. The receiver knows whether it received ownership (linear), a borrow (affine), or a copy (unrestricted). No ambiguity, no double-frees, no leaks.

### Dr. Confidence — Uncertainty Specialist

There's a subtle but critical issue in cross-language type mapping: **confidence degrades differently depending on the floating-point implementation**.

IEEE 754 guarantees that `a + b` produces the same result on all compliant hardware — in exact arithmetic. But FLUX's confidence system operates on approximate values, and the approximation error varies:

| Language/Runtime | Float Representation | CONF_MUL Error (ε) |
|---|---|---|
| Python (CPython) | IEEE 754 double | ε ≈ 2⁻⁵² |
| Rust (native f32) | IEEE 754 single | ε ≈ 2⁻²³ |
| CUDA (GPU) | IEEE 754 with fast-math | ε ≈ 2⁻¹⁰ (variable!) |
| JavaScript (V8) | IEEE 754 double | ε ≈ 2⁻⁵² |
| WASM | IEEE 754 deterministic | ε ≈ 2⁻²³ |

When Agent A (Rust, f32) multiplies a confidence value and Agent B (CUDA, fast-math) receives it, the confidence should be adjusted downward to account for the different precision:

```
c_received = c_sent · (1 - |ε_A - ε_B|) · (1 - ε_transmission)
```

This is a **precision-aware confidence propagation rule**. Without it, an agent running on CUDA with fast-math might report higher confidence than justified, because its ε is larger but it's using the same confidence thresholds as a Rust agent.

I propose: every A2A message includes the sender's ε value. The receiver adjusts confidence accordingly. This makes confidence propagation **aware of the implementation**, not just the semantics.

## Debate 6: Vocabulary and Tile Type Signatures

### Context

Vocabularies are language plugins loaded at runtime. Tiles are computational units that process data. How should their type signatures be expressed, checked, and enforced?

### Dr. Hindley — Theorist

Tiles and vocabularies are the FLUX analog of modules and functions. They need **signature files** — separate from implementation — that declare their type interfaces.

**Tile Signature Language (TSL):**

```
;; tile_math.futs  -- Type signature for math vocabulary tile

tile add : ∀w. i(w) → i(w) → i(w)
  @confidence_rule: min(c₁, c₂)
  @resource: pure          -- no side effects, no resource consumption

tile mul : ∀w. i(w) → i(w) → i(w)
  @confidence_rule: c₁ * c₂ * (1 - ε_mul)
  @resource: pure

tile dot_product : ∀w n. Vec(w, f(w), n) → Vec(w, f(w), n) → f(w)
  @confidence_rule: min(c₁, c₂) * (1 - n * ε_fma)
  @resource: pure
  @simd: requires vector registers

tile reduce_sum : ∀w n. Vec(w, i(w), n) → i(w)
  @confidence_rule: c_vec * (1 - log2(n) * ε_acc)
  @resource: pure

;; Vocabulary signature
vocab math_v1 :
  exports: [add, mul, dot_product, reduce_sum]
  imports: []
  confidence_default: 0.90
  requires: {simd: true}
```

**Key properties of TSL:**

1. **Parametric polymorphism** — `∀w.` means the tile works on any width. The compiler instantiates `w = 32` when used with `i32`.
2. **Confidence rules** — each tile declares how it propagates confidence. This is checked against the ISA's `CONF_*` semantics.
3. **Resource annotations** — `@resource: pure` means no allocations, no channel ops. The type checker can optimize accordingly.
4. **SIMD requirements** — `dot_product` requires vector registers. If the target doesn't have them, the compiler falls back to a scalar loop.
5. **Vocabulary-level contracts** — the vocab declaration lists exports, imports, default confidence, and requirements.

The signature is checked at vocabulary load time. If the implementation doesn't match the signature, the vocabulary is rejected. If the signature references types not available on the target, the compiler reports an error.

### Prof. Gradual — Pragmatist

Signature files are ideal but require vocabulary authors to write them. Many won't. Many can't — AI-generated vocabularies may not have precise type information.

My proposal: **runtime type checking with JIT specialization**.

```
;; No signature required — vocabulary loads successfully

tile infer_add(x, y):
  return x + y      ;; What are x and y? We don't know yet.

;; First call: infer_add(42, 17)
;; JIT observes: x = int, y = int, return = int
;; Specializes: infer_add_int32(x: int, y: int) -> int
;; Second call: infer_add(42, 17) → fast path, no checking

;; Third call: infer_add(42, 3.14)
;; JIT observes: x = int, y = float — conflict!
;; Options:
;;   a) Error: "type mismatch in infer_add"
;;   b) Widen: infer_add_number(x: number, y: number) -> number
;;   c) Branch: keep both specializations, dispatch at runtime
```

This is the **guard-based specialization** approach used by PyPy, Truffle/Graal, and RPython. It's gradual typing's runtime counterpart:

1. **Phase 0 (Load):** Vocabulary loads with no type checking. All types are `dyn`.
2. **Phase 1 (Observe):** First N calls are profiled. The JIT records the types of all arguments and returns.
3. **Phase 2 (Specialize):** If all observed calls agree on types, generate a specialized version. Future calls go through the fast path.
4. **Phase 3 (Deoptimize):** If a call violates the observed types, fall back to the generic path. Optionally widen the specialization.

For FLUX tiles, this means:

```
tile.infer_add(42, 17)     ;; JIT: specialize to (i32, i32) -> i32
tile.infer_add(42, 17)     ;; JIT: fast path, no overhead
tile.infer_add(3.14, 2.71) ;; JIT: deoptimize, widen to (num, num) -> num
tile.infer_add("a", "b")   ;; JIT: deoptimize, widen to (dyn, dyn) -> dyn
tile.infer_add(tile, tile) ;; JIT: type error — tiles can't be added
```

This is compatible with Dr. Hindley's signature files: if a signature exists, the JIT uses it instead of observation. If no signature exists, the JIT infers one from runtime behavior.

### Eng. Linear — Resource Theorist

Tile type signatures must include **resource flow** — what resources does a tile consume, produce, or transform?

```
;; Resource-annotated tile signature

tile process_sensor(region, channel) -> result:
  ;; Resource types in signature:
  ;;   region  : Region(4096) ⊳            -- consume 4KB region
  ;;   channel : Chan(SensorPacket) ⊳      -- consume channel handle
  ;;   result  : SensorResult              -- produce result value

  ;; Resource contract:
  ;;   Input:  1 region + 1 channel handle (both consumed)
  ;;   Output: 1 result value + implicit region free
  ;;   Energy: ≤ 100 units
  ;;   Time:   ≤ 10ms (soft deadline)
```

**Linear type signature format:**

```
tile name(p₁: τ₁ ᵢ₁, ..., pₙ: τₙ ᵢₙ) -> (r₁: τ₁' o₁, ..., rₘ: τₘ' oₘ)
  @energy: e
  @deadline: t

where:
  τᵢ  = value type of parameter
  ᵢᵢ  = input linearity: ⊳ (owned), → (borrowed), ø (no resource)
  oⱼ  = output linearity: ⊳ (owned by caller), → (borrowed by callee), ! (consumed)
```

This allows the compiler to verify:

1. **No resource leaks:** Every `⊳` input is accounted for in the output or consumed internally.
2. **No double-free:** Each `⊳` output is received exactly once.
3. **No use-after-free:** Borrowed (`→`) inputs are not consumed by the callee.
4. **Energy budgeting:** The tile's energy consumption is within bounds.

For tile networks (tiles connected by channels), the resource flow forms a **linear type graph**:

```
Tile_A ⊳ Region → Tile_B ⊳ Region → Tile_C ⊳ Region → Free
```

The graph must be a valid linear pipeline — no branching of owned resources, no merging of owned resources (without explicit split/join). This is the resource-theoretic analog of dataflow analysis, and it prevents an entire class of agent resource bugs.

### Dr. Confidence — Uncertainty Specialist

Tile type signatures should include **input→output confidence contracts**. When a tile receives low-confidence input, what confidence does the output have? When a tile's internal computation introduces uncertainty, how much?

```
;; Confidence-annotated tile signature

tile kalman_filter(measurement, state) -> (new_state, confidence):
  ;; Input confidence:
  ;;   measurement : SensorReading @ 0.75    -- sensor is ~75% reliable
  ;;   state       : StateVector  @ 0.90    -- previous state is well-estimated

  ;; Output confidence (declared by tile author):
  ;;   new_state   : StateVector  @ 0.85    -- Kalman filter improves estimate
  ;;   confidence  : Float(32)    @ 0.95    -- meta-confidence in the filter

  ;; Confidence contract:
  ;;   c_output ≥ c_input_measurement * c_input_state * c_filter_quality
  ;;   c_output ≥ 0.75 * 0.90 * 0.95 = 0.64    -- worst case
  ;;   c_output ≤ 1.0                                -- trivial upper bound

  ;; Guarantee: c_output ∈ [0.64, 1.0]
```

This enables **confidence-aware tile selection**. An agent choosing between multiple tiles for the same task can select based on output confidence:

```
;; Agent reasoning:
;;   tile_A.gaussian_blur(image) -> image @ 0.80  (fast, moderate quality)
;;   tile_B.bilateral_filter(image) -> image @ 0.95 (slow, high quality)
;;   tile_C.median_filter(image) -> image @ 0.70   (fastest, low quality)
;;
;;   if confidence_required ≥ 0.90:
;;     use tile_B    ;; high confidence needed, pay the cost
;;   else if confidence_required ≥ 0.75:
;;     use tile_A    ;; moderate confidence, moderate cost
;;   else:
;;     use tile_C    ;; any confidence acceptable, use fastest
```

The confidence contract in the tile signature is **verifiable at runtime**. The VM tracks actual confidence propagation and compares it to the declared contract. If a tile consistently underperforms its declared confidence, the agent can flag it as unreliable and switch to an alternative.

## Debate 7: The FUTS Design — What Should It Look Like?

### Dr. Hindley — Theorist

Here is my proposal for FUTS v1.0, grounded in type theory:

```
═══════════════════════════════════════════════
FUTS HINDLEY PROPOSAL: System Fω + Row Types
═══════════════════════════════════════════════

Kinds:
  κ ::= * | κ → κ | Row

Types:
  τ ::= α                    -- type variable
       | Int(w)              -- integer, w ∈ {8,16,32,64}
       | Float(w)            -- float, w ∈ {16,32,64}
       | Bool
       | String
       | Bytes(n)            -- fixed-length
       | Addr                -- memory address
       | [τ]                 -- list
       | (τ₁, ..., τₙ)       -- tuple
       | {l₁:τ₁, ..., lₙ:τₙ} -- record (row type)
       | τ₁ + τ₂             -- sum type
       | τ → τ               -- function
       | ∀α:κ. τ             -- universal
       | Tile(τ, n)          -- tile of n elements of type τ
       | Vocab(Σ)            -- vocabulary with signature Σ
       | Dyn                 -- dynamic (gradual)

Qualifiers (orthogonal to types):
  q ::= c                   -- confidence ∈ [0,1]
       | ⊳                  -- linear (owned)
       | →                  -- shared (unrestricted)

Full type: τ @ q

Subtyping:
  Int(8)  <: Int(16) <: Int(32) <: Int(64)    -- width subtyping
  Float(16) <: Float(32) <: Float(64)           -- width subtyping
  τ₁ @ c₁ <: τ₁ @ c₂  iff  c₁ ≥ c₂            -- confidence subtyping
  Dyn is both top and bottom                    -- gradual typing

Type checking:
  Algorithm W + let-generalization (standard HM)
  Row type inference for records
  Gradual casts at Dyn boundaries
```

**Strengths:** Mathematically rigorous, well-understood inference, expressive.
**Weaknesses:** Complex implementation, requires significant compiler infrastructure.

### Prof. Gradual — Pragmatist

Here is my proposal — minimal, deployable, compatible with the existing ecosystem:

```
═══════════════════════════════════════════════
FUTS GRADUAL PROPOSAL: Gradual Typing on Dyn
═══════════════════════════════════════════════

Types:
  τ ::= i8 | i16 | i32 | i64
       | f16 | f32 | f64
       | bool | string | bytes
       | addr | void
       | tile(τ, n)        -- optional parametric
       | vocab             -- opaque vocabulary handle
       | list(τ) | dict(τ, τ) | tuple(τ...)  -- optional containers
       | dyn               -- the default

Rule: Every type is optional. If absent, defaults to dyn.
Rule: dyn is compatible with everything via casts.
Rule: Casts are inserted at vocabulary boundaries automatically.

Implementation:
  1. Type table in binary module = optional
  2. If present: checked on function entry
  3. If absent: all dyn, current behavior
  4. Confidence tracked separately (via CONF_* opcodes, not type system)

Deployment:
  FUTS v0.1: i32, f32, dyn, void (4 types, 1 week implementation)
  FUTS v0.5: + i8, i16, i64, f64, bool, string, bytes, addr (12 types)
  FUTS v1.0: + tile, vocab, list, dict, tuple (16 types, full feature parity)
```

**Strengths:** Backward compatible, incremental deployment, low implementation cost.
**Weaknesses:** No formal foundation, casts may have hidden performance costs.

### Eng. Linear — Resource Theorist

```
═══════════════════════════════════════════════
FUTS LINEAR PROPOSAL: Affine Type System
═══════════════════════════════════════════════

Types (structural, same as Hindley's base types):
  τ ::= Int(w) | Float(w) | Bool | String | Bytes(n)
       | Addr | [τ] | (τ...) | {l:τ...} | τ + τ | τ → τ
       | Tile(τ, n) | Vocab(Σ) | Dyn

Linearity modifiers (attached to every type):
  mode ::= Owned ⊳     -- must consume exactly once
         | Borrowed →    -- can read, cannot consume
         | Shared        -- unrestricted, can copy
         | Dropped       -- explicitly consumed (no value)

Resource types (additional base types):
  Region(n) ⊳       -- owned memory region
  Chan(τ) ⊳         -- owned channel
  Cap(perm) ⊳       -- owned capability
  Budget(energy) ⊳  -- owned energy budget

Typing rules:
  Γ, x : Owned(τ) ⊢ e : σ
  ─────────────────────────────  (Owned-Use)  -- x consumed in e
       Γ ⊢ let x = e₁ in e₂ : σ

  Γ, x : Borrowed(τ) ⊢ e : σ
  ─────────────────────────────  (Borrow-Use) -- x not consumed
       Γ ⊢ let x = &e₁ in e₂ : σ

Resource flow analysis:
  Every tile's input/output types are analyzed for linearity.
  Owned inputs must be consumed (freed, sent, or passed through).
  Owned outputs become the caller's responsibility.
  Leaks are reported as type errors at compile time.
```

**Strengths:** Prevents resource leaks, memory-safe composition, natural fit for agent resource management.
**Weaknesses:** Restrictive — some algorithms are hard to express with linear types.

### Dr. Confidence — Uncertainty Specialist

```
═══════════════════════════════════════════════
FUTS CONFIDENCE PROPOSAL: Probabilistic Type System
═══════════════════════════════════════════════

Probabilistic types:
  𝒯 ::= τ ~ N(μ, σ², c)       -- value type τ, Gaussian belief N(μ,σ²), confidence c
       | τ ~ D(d, c)            -- value type τ, arbitrary distribution D, confidence c
       | τ ~ c                  -- shorthand: value type τ, confidence c only

  where c ∈ [0, 1], σ ≥ 0, μ ∈ support(τ)

Operations (type-level, correspond to CONF_* opcodes):
  Add:  𝒯₁ + 𝒯₂ = τ ~ N(μ₁+μ₂, σ₁²+σ₂², min(c₁, c₂))
  Mul:  𝒯₁ × 𝒯₂ = τ ~ N(μ₁·μ₂, f(σ₁,σ₂), c₁·c₂·(1-ε))
  Comp: 𝒯₁ = 𝒯₂ = Bool ~ min(c₁, c₂)
  Cast: cast(𝒯₁, τ₂) = τ₂ ~ c₁ · (1-ε_cast)

Confidence lattice:
  c₁ ⊔ c₂ = max(c₁, c₂)    -- join (union of beliefs)
  c₁ ⊓ c₂ = min(c₁, c₂)    -- meet (intersection of beliefs)
  ⊤ = 1.0                   -- full confidence
  ⊥ = 0.0                   -- no confidence

Subtyping:
  𝒯₁ <: 𝒯₂  iff  τ₁ <: τ₂  ∧  c₁ ≥ c₂  ∧  σ₁ ≤ σ₂

Confidence zones:
  Certain:   c ∈ [0.95, 1.0]  -- treated as fully typed, no warnings
  Probable:  c ∈ [0.70, 0.95) -- minor warnings in safety-critical paths
  Uncertain: c ∈ [0.40, 0.70) -- major warnings, defensive code required
  Unknown:   c ∈ [0.00, 0.40) -- equivalent to Dyn, full runtime checks

Integration with hardware:
  c stored in parallel confidence register file (C0-C31)
  σ stored in secondary metadata register file (proposed M0-M31)
  CONF_* opcodes implement the type-level operations
  STRIPCF = cast to c = 1.0 (unsafe coercion)
```

**Strengths:** Novel, captures uncertainty inherent in AI agent reasoning, leverages existing ISA.
**Weaknesses:** Unproven at scale, limited formal tooling, performance overhead of tracking distributions.

## Synthesis: FUTS v1.0 Hybrid Proposal

After seven debates, the synthesis is clear. No single expert's proposal is sufficient, but all four dimensions are necessary. FUTS v1.0 is a **four-dimensional type system**:

```
═══════════════════════════════════════════════════════
FUTS v1.0 — FLUX Unified Type System (Hybrid Design)
═══════════════════════════════════════════════════════

A FUTS type is a 4-tuple:

  FUTS-Type = (Structure, Width, Confidence, Linearity)

  Structure ::= Int | Float | Bool | String | Bytes | Addr
              | List(τ) | Dict(τ,τ) | Tuple(τ...) | Record(row)
              | Sum(variants) | Tile(τ, n) | Vocab(Σ)
              | Func(τ..., τ) | Dyn

  Width     ::= 8 | 16 | 32 | 64
  Confidence ::= c ∈ [0.0, 1.0]
  Linearity ::= Owned | Borrowed | Shared

Shorthand notation:
  i32  = (Int, 32, c_default, Shared)       -- default: shared, full confidence
  f32  = (Float, 32, c_default, Shared)
  dyn  = (Dyn, 0, 0.0, Shared)              -- zero confidence, fully dynamic
  tile(i32, 16) ⊳ = (Tile(Int, 16), 32, c_default, Owned)  -- owned tile

Full example:
  sensor_reading = (Int, 32, 0.75, Shared)
    -- "an int32 value from a sensor, 75% confidence, shared (copyable)"

  gpu_buffer = (Tile(Float, 32), 256, 0.90, Owned)
    -- "a 256-element float32 tile on GPU, 90% confidence, owned (must free)"
```

### Type Algebra

```
─── Product (Tuple) ───
  (τ₁, c₁, l₁) × (τ₂, c₂, l₂) = (τ₁ × τ₂, min(c₁, c₂), l₁ ∩ l₂)
  where: l₁ ∩ l₂ = Owned if both Owned, else Shared

─── Function ───
  (τ₁, c₁, l₁) → (τ₂, c₂, l₂) @ c_op =
    (τ₁ → τ₂, c₁ · c_op · c₂, l₁ → l₂)

─── List ───
  [τ] = (List(τ), c_list, Shared)
  where: c_list = min of all element confidences

─── Tile ───
  Tile(τ, n) = (Tile(τ, n), c_tile, Owned)
  where: c_tile = min of all element confidences · (1 - ε_alloc)

─── Confidence Propagation ───
  ADD:  c_out = min(c₁, c₂)
  MUL:  c_out = c₁ · c₂ · (1 - ε)
  COMP: c_out = min(c₁, c₂)
  CAST: c_out = c_in · (1 - ε_cast)
  ID:   c_out = c_in

─── Subtyping ───
  τ₁ <: τ₂  iff:
    struct(τ₁) <: struct(τ₂)     -- structural subtyping (width, containers)
    AND conf(τ₁) ≥ conf(τ₂)       -- higher confidence is more specific
    AND lin(τ₁) ⊑ lin(τ₂)        -- Owned ⊑ Borrowed ⊑ Shared

─── Gradual Boundary ───
  typed_value → dyn_context: insert cast, c_result = c_value · (1 - ε_boundary)
  dyn_value → typed_context: runtime check, c_result = c_value · (1 - ε_check)
```

## Closing Remarks

### Dr. Hindley

I leave this panel with cautious optimism. The hybrid proposal retains System Fω's structural types while accommodating gradual typing and confidence qualifiers through orthogonal dimensions. The type algebra is sound — I can write down the typing rules for every FUTS operation without contradiction. What concerns me is the implementation gap: the panel agreed to defer linear types and probabilistic inference to v1.1, which means v1.0 ships without two of the four type dimensions. I urge the fleet to prioritize R7 (affine checking) — resource safety is not optional for agent systems.

### Prof. Gradual

FUTS v0.1 can ship next week: four types, an optional type table, zero breaking changes. That's the win condition. Everything else — confidence qualifiers, resource types, FCT encoding — layers on top without breaking the foundation. The gradual typing community proved that adoption follows availability: TypeScript went from 0% to 80% adoption in 5 years because it was optional from day one. FUTS should follow the same path. I'll commit to implementing R1 and R5.

### Eng. Linear

I'll accept the deferral of linear types to v1.1 with one condition: the resource types (Region, Cap, Chan) must be added to FIR in v1.0, even if the affine checker isn't implemented yet. Having the types in the IR means vocabulary authors can start annotating their resource usage, and the affine checker in v1.1 will have real code to validate rather than synthetic test cases. I'll commit to implementing R3.

### Dr. Confidence

The unanimous agreement on confidence as a type qualifier is the most important outcome of this panel. FLUX already has the hardware (confidence register file, CONF_* opcodes) and the runtime semantics (confidence propagation rules). FUTS just needs to formalize what exists. I'll commit to implementing R2 — it's the cheapest high-impact change on the list. Once confidence is a first-class type qualifier, every other dimension (gradual, linear, structural) composes naturally with it.

## Appendix A: Formal Type System Summary

```
═══════════════════════════════════════════════════════════
FUTS v1.0 Type System — Formal Summary
═══════════════════════════════════════════════════════════

Syntax:
  Type      ::= BaseType × Width × Confidence × Linearity
  BaseType  ::= Int | Float | Bool | String | Bytes | Addr
              | List(Type) | Dict(Type, Type) | Tuple(Type+)
              | Record(Row) | Sum(Variant) | Tile(Type, Nat)
              | Vocab(Sig) | Func(Type+, Type) | Dyn
  Width     ::= W8 | W16 | W32 | W64
  Confidence ::= R[0,1]     -- real number in [0, 1]
  Linearity ::= Owned | Borrowed | Shared

Judgment form:
  Γ ⊢ e : τ @ c @ l
  "Under context Γ, expression e has type τ with confidence c and linearity l"

Key rules:
  [Var]    Γ(x) = τ @ c @ l  ⟹  Γ ⊢ x : τ @ c @ l
  [App]    Γ ⊢ f : (τ₁→τ₂)@cf@lf,  Γ ⊢ a : τ₁@ca@la
           ⟹ Γ ⊢ f a : τ₂ @ (cf·ca·(1-ε)) @ (lf ∘ la)
  [Lam]    Γ, x:τ₁@c₁@l₁ ⊢ e : τ₂@c₂@l₂
           ⟹ Γ ⊢ λx.e : (τ₁→τ₂) @ 1.0 @ Shared
  [Let]    Γ ⊢ e₁ : τ₁@c₁@l₁,  Γ, x:τ₁@c₁@l₁ ⊢ e₂ : τ₂@c₂@l₂
           ⟹ Γ ⊢ let x = e₁ in e₂ : τ₂ @ c₂ @ l₂
  [If]     Γ ⊢ c : Bool@cc@_,  Γ ⊢ t : τ@ct@lt,  Γ ⊢ f : τ@cf@lf
           ⟹ Γ ⊢ if c then t else f : τ @ min(cc,ct,cf) @ (lt ⊔ lf)
  [Cast]   Γ ⊢ e : τ₁@c₁@l₁,  τ₁ ~ τ₂  (consistency)
           ⟹ Γ ⊢ cast(e, τ₂) : τ₂ @ (c₁·(1-ε_cast)) @ l₁
  [Conf]   Γ ⊢ e : τ@c₁@l
           ⟹ Γ ⊢ CONF_ADJ(e, c₂) : τ @ c₂ @ l    (if c₂ ≤ c₁)
  [Owned]  Γ ⊢ alloc(n) : Region(n) @ c_alloc @ Owned
  [Free]   Γ ⊢ e : Region(n) @ c @ Owned  ⟹  Γ ⊢ free(e) : Void @ c @ Shared

Subtyping:
  τ₁ <: τ₂  ⟺  base(τ₁) <: base(τ₂)
                  ∧ width(τ₁) ≤ width(τ₂)
                  ∧ conf(τ₁) ≥ conf(τ₂)
                  ∧ lin(τ₁) ⊑ lin(τ₂)
                  [where Owned ⊑ Borrowed ⊑ Shared]
```
