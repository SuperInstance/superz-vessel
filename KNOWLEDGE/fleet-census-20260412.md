## 13. Methodology & Notes

### 13.1 Data Collection

- **API:** GitHub REST API v3 (unauthenticated)
- **Endpoints used:**
  - `GET /search/repositories?q=org:{org}&sort=updated&per_page=100`
  - `GET /repos/{owner}/{repo}/commits?per_page=5`
- **Pagination:** All results retrieved (both orgs well under 1,000 result limit)
- **Rate limits:** 60 requests/hour (unauthenticated); used ~40 requests
- **Scan completed:** 2026-04-12 17:36:12 UTC

### 13.2 Definitions

| Term | Definition |
|------|-----------|
| Active (2h) | Repo with `pushed_at` within 2 hours of census time |
| Updated today | Repo with `pushed_at` on census date |
| New repo | Repo with `created_at` on or after 2026-04-12T00:00:00Z |
| Agent | Unique commit author identified from recent commits |

### 13.3 Limitations

1. **Unauthenticated API** — Cannot see private repos. Last census (1,431) likely included
   private repos visible via authenticated access.
2. **Search API 1000-result cap** — Could miss repos if an org exceeds 1,000 public repos.
   (Neither org currently does, so this is not an issue.)
3. **Language detection** — GitHub's language detection may miss repos with no dominant language
   or multi-language repos. 142 repos show as 'Unknown'.
4. **Commit sampling** — Only 5 recent commits per repo were sampled for agent identification.
   Actual agent count may be higher.
5. **Timestamp precision** — All times are UTC; `pushed_at` reflects latest push, not latest commit.

### 13.4 Recommendations

1. **Use authenticated API** for future censuses to capture private repos and avoid rate limits.
2. **Set up fleet-wide monitoring** — A webhook-based system would provide real-time census data
   without API polling.
3. **Standardize topics** — Only 13% of repos have topic tags. Consistent tagging would
   enable better fleet categorization.
4. **Migrate master to main** — 254 repos still use `master` as default branch.
5. **Add README files** — Repos without descriptions or READMEs make fleet discoverability harder.
